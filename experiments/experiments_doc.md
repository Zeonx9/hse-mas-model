# Система экспериментов музейного комплекса

## Принцип работы

Система экспериментов реализует подход, аналогичный NetLogo BehaviorSpace: автоматизированный прогон модели с различными комбинациями входных параметров и сбор итоговых метрик в CSV-файл для последующего анализа.

### Архитектура

Система состоит из двух компонентов:

1. **Python-оркестратор** (`experiments/run_experiments.py`) — читает JSON-конфиг, генерирует все комбинации параметров, для каждой запускает JVM с моделью и ждёт завершения.
2. **Experiment mode в MuseumEnv.java** — среда Jason-модели, которая при наличии параметров `maxDays` и `csvFile` работает в ускоренном режиме: без визуальных задержек, с автоматической остановкой по достижении заданного дня и записью строки результатов в CSV.

### Цикл одного эксперимента

1. Скрипт генерирует временный `museum.mas2j` с нужными параметрами среды и числом агентов-посетителей.
2. Запускается `gradle run -q` (тихий режим, без лишнего вывода Gradle).
3. Модель работает без `Thread.sleep` — агенты принимают решения на каждом шаге, шаги выполняются с минимальной задержкой.
4. По достижении `maxDays` среда записывает одну строку в CSV и останавливает MAS.
5. Процесс завершается, скрипт переходит к следующему эксперименту.
6. После всех прогонов оригинальный `museum.mas2j` восстанавливается.

---

## Запуск

Из корневой директории проекта (`museum_complex/`):

```bash
# С конфигом по умолчанию (experiments/experiment.json)
python3 experiments/run_experiments.py

# С произвольным конфигом
python3 experiments/run_experiments.py experiments/my_config.json
```

Требования: Python 3.9+, Gradle (доступен в PATH), собранный проект (`gradle build`).

## Конфигурация экспериментов (JSON)

Файл конфигурации задаёт все параметры серии экспериментов.

### Структура

```json
{
  "maxDays": [365, 730, 1095],
  "runsPerCombination": 3,
  "numVisitors": 65,
  "output": "experiments/results.csv",
  "parameters": {
    "museumCapacity": [10, 15, 20],
    "hotelCapacity": [5, 7],
    "ticketPrice": [20, 30, 50],
    "hotelPrice": [30, 50],
    "monthlyExpenditures": [5000],
    "investMinProb": [0.1, 0.8]
  }
}
```

### Описание полей

| Поле | Тип | Описание |
|---|---|---|
| `maxDays` | int или int[] | Длительность прогона в модельных днях. Одно значение или массив для перебора (масштабирование по времени) |
| `runsPerCombination` | int | Сколько раз прогонять каждую комбинацию параметров (для статистической устойчивости, т.к. модель стохастическая) |
| `numVisitors` | int | Число агентов-посетителей (по умолчанию 65, если не указано) |
| `output` | string | Путь к выходному CSV-файлу (относительно корня проекта) |
| `parameters` | object | Словарь параметров, каждый содержит массив значений для перебора |

`maxDays` участвует в декартовом произведении наравне с параметрами из `parameters`. Это позволяет анализировать поведение модели при масштабировании по времени.

### Параметры модели

| Параметр | Описание | Типичные значения |
|---|---|---|
| `investMinProb` | Нижняя граница вероятности инвестирования менеджером при дефиците ремонтного фонда (см. `manager.asl`, приоритет 4) | 0.0–0.5 |
| `museumCapacity` | Максимальное число посетителей музея в день | 10, 15, 20, 30 |
| `hotelCapacity` | Число мест в гостинице | 3, 5, 7, 10 |
| `ticketPrice` | Цена входного билета | 20, 30, 50, 100 |
| `hotelPrice` | Цена за ночь в гостинице | 30, 50, 80 |
| `monthlyExpenditures` | Ежемесячные расходы на содержание (электричество, зарплаты и т.д.) | 3000, 5000, 7000 |

### Расчёт числа экспериментов

Общее число = (произведение длин всех массивов в `parameters`) x (число значений `maxDays`) x `runsPerCombination`.

Пример: `[10,15,20] x [3,5,7] x [20,30,50] x [30,50] x [5000]` x `[365,730,1095]` = 54 x 3 = 162 комбинации x 3 прогона = **486 экспериментов**.

## Формат выходного CSV

Каждый прогон добавляет одну строку в CSV. Заголовок записывается автоматически при создании файла.

### Столбцы

#### Входные параметры (8 столбцов)

| Столбец | Описание |
|---|---|
| `museumCapacity` | Вместимость музея |
| `hotelCapacity` | Вместимость гостиницы |
| `ticketPrice` | Цена билета |
| `hotelPrice` | Цена гостиницы |
| `monthlyExpenditures` | Ежемесячные расходы |
| `investMinProb` | Нижняя граница вероятности инвестирования менеджером при дефиците ремонтного фонда |
| `numVisitors` | Число агентов-посетителей |
| `maxDays` | Длительность прогона (дней) |

#### Итоговые метрики (14 столбцов)

| Столбец | Описание |
|---|---|
| `totalVisits` | Суммарное число визитов за весь прогон |
| `totalHotelStays` | Суммарное число проживаний в гостинице |
| `totalRefusals` | Суммарное число отказов (не пошёл / не смог) |
| `totalRepairs` | Суммарное число дней ремонта |
| `finalWear` | Износ здания на конец симуляции (%) |
| `finalAttractiveness` | Привлекательность на конец симуляции (0-100) |
| `finalBudget` | Бюджет на конец симуляции |
| `finalMobileNetwork` | Состояние мобильной сети (0-100) |
| `finalPaymentSystem` | Состояние платёжной системы (0-100) |
| `finalTransportAccess` | Транспортная доступность (0-100) |
| `finalInternetQuality` | Качество интернета (0-100) |
| `finalNavigationAccess` | Навигационная доступность (0-100) |
| `finalServiceAvailability` | Доступность сервисов (0-100) |
| `finalAvgReview` | Средняя оценка посетителей (0-100) |

## Информация о модели

### Агенты

| Агент | Количество | Роль |
|---|---|---|
| `visitor` | Настраивается (по умолчанию 65) | Потенциальный посетитель музея. Каждый день проходит 3-ступенчатую проверку: транспорт -> оплата -> желание. Оставляет отзыв при визите. |
| `manager` | 1 | Управляющий комплексом. Принимает решения об инвестициях и ремонте на основе состояния среды и бюджета. Ведёт переговоры с реставратором. |
| `restorer` | 1 | Реставратор. Имеет собственную загрузку и доступность. Отвечает на запросы менеджера котировками, выполняет ремонт (снижает wear на 1%/день). |

### Ключевые механики

- **3-ступенчатая воронка посетителя**: P(визит) = P(добрался) x P(оплатил) x P(захотел). Каждая стадия зависит от соответствующего инфраструктурного фактора.
- **6 факторов инфраструктуры**: мобильная сеть, платёжная система, транспорт, интернет, навигация, сервис. Деградируют со временем и от нагрузки. Менеджер инвестирует в самый слабый.
- **Система отзывов**: посетители оценивают визит; отзывы влияют на привлекательность (экспоненциальное сглаживание с весом 0.2).
- **Переговоры о ремонте**: менеджер запрашивает котировку, реставратор может отказать или назвать цену в зависимости от загрузки. Оплата 50% авансом, 50% по завершении.
- **Ежемесячные расходы**: фиксированная сумма списывается каждые 30 дней.
- **Сезонность**: зима (SF=0.3), весна (0.7), лето (1.0), осень (0.5) — влияет на желание посетителей.

### Свободный прогон vs Эксперимент

| | Свободный прогон | Эксперимент |
|---|---|---|
| Запуск | `gradle run` | `python3 experiments/run_experiments.py` |
| Задержка между днями | 200мс | Нет |
| Остановка | Ручная (закрытие окна / Ctrl+C) | Автоматическая по `maxDays` |
| MAS Console | Открывается | Открывается (вывод подавлен `-q`) |
| CSV-выход | Нет | Одна строка за прогон |
| Конфигурация | `museum.mas2j` | JSON-конфиг -> генерируемый `museum.mas2j` |

### Оценка времени выполнения

Один прогон 365 дней с 65 посетителями в experiment mode занимает ~2-5 секунд (зависит от машины). Время линейно зависит от `maxDays`: 730 дней ~4-10с, 1095 дней ~6-15с. Для 486 экспериментов (с тремя горизонтами): ~30-60 минут.
