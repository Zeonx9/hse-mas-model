# Museum Complex -- BDI Multi-Agent Simulation

Многоагентная BDI-модель музейно-архитектурного комплекса, реализованная на платформе Jason (AgentSpeak).
Моделирует ежедневную работу музея: посещения, управление бюджетом, деградацию инфраструктуры, ремонт и сезонную динамику.

Поддерживает два режима работы менеджера:
- **Классический** -- хардкод-логика принятия решений на ASL (правила приоритетов).
- **AI-powered** -- стратегические решения делегируются LLM (qwen/qwen3-coder-next) через OpenRouter API. Механические действия (исполнение ремонта, auto-accept котировки, таймауты) остаются хардкодом в ASL.

## Структура проекта

```

  src/
    agt/

    java/
      museum/
```

## Требования

- **Java 17+**
- **Gradle 8+**
- **Jason 3.3.0** -- интерпретатор Jason (бинарная сборка `jason-bin-3.3.0`)
- **Python 3.9+** -- для запуска экспериментов
- **Переменная окружения `OPENROUTER_API_KEY`** -- для AI-менеджера (без ключа менеджер делает `skip`)

Зависимость от Jason подключена через Maven-репозиторий в `build.gradle`, отдельная установка бинарника не требуется для сборки и запуска через Gradle.

## Запуск

### Свободный прогон (интерактивный)

```bash
gradle run                              # без AI (skip при отсутствии ключа)
OPENROUTER_API_KEY=sk-... gradle run    # с AI-менеджером
```

Открывается MAS Console с логами. Симуляция работает бесконечно (дни идут с задержкой 200мс). Для остановки -- закрыть окно консоли или нажать Ctrl+C.

В логах AI-решения видны как: `[manager] Day N: AI decision -> invest_infra mobile_network`.

### Сборка проекта

```bash
gradle build
```

### Запуск экспериментов

```bash
python3 experiments/run_experiments.py                            # конфиг по умолчанию
python3 experiments/run_experiments.py experiments/my_config.json  # свой конфиг
```

Подробнее -- см. `experiments/experiments_doc.md`.

## Конфигурация модели

## Среда (MuseumEnv)

Среда реализована как `TimeSteppedEnvironment` -- один шаг = один модельный день.

### Состояние среды

- **Бюджет** -- пополняется от продажи билетов и проживания, расходуется на ремонт, инвестиции, ежемесячное содержание.
- **Износ (wear)** -- растёт ежедневно: базовый +0.05%/день, +0.02% за каждого посетителя, 5% шанс случайного повреждения (+0.5..2.0%).
- **Привлекательность (attractiveness)** -- корректируется отзывами посетителей (экспоненциальное сглаживание, вес 0.2).
- **6 факторов инфраструктуры** -- деградируют двумя способами:
  - Нагрузочная деградация (мобильная сеть, платёжная система) -- пропорционально числу посетителей.
  - Временная деградация (транспорт, интернет, навигация, сервис) -- фиксированная -0.03/день.
- **Сезонность** -- сезонный фактор (SF) влияет на желание посетителей:
  - Зима (дни 0-59, 335-364): SF = 0.3
  - Весна (60-151): SF = 0.7
  - Лето (152-243): SF = 1.0
  - Осень (244-334): SF = 0.5
- **Ремонт** -- пока `repairing = true`, вместимость музея сокращается вдвое.

### Действия агентов в среде

| Действие | Агент | Эффект |
|---|---|---|
| `visit(yes/no, Review)` | visitor | +ticketPrice к бюджету, +hotelPrice если hotel=yes, запись отзыва |
| `refuse_visit` | visitor | Счётчик отказов +1 |
| `invest_attractiveness` | manager | -500 бюджета, +10 привлекательности |
| `invest_infra(Factor)` | manager | -500 бюджета, +10 к указанному фактору |
| `order_repair(Price)` | manager | 50% цены авансом, запуск ремонта |
| `do_repair` | restorer | wear -1%/день |
| `skip` | любой | Пустое действие (обязательно для синхронизации TSE) |

### Перцепты (доступны всем агентам)

`step(Day)`, `season(S)`, `season_factor(SF)`, `wear(W)`, `attractiveness(A)`, `museum_price(P)`, `hotel_price(P)`, `museum_slots_free(N)`, `hotel_rooms_free(N)`, `mobile_network(V)`, `payment_system(V)`, `transport_access(V)`, `internet_quality(V)`, `navigation_access(V)`, `service_availability(V)`, `avg_review(V)`, `repairing(yes/no)`.

Менеджер дополнительно получает: `budget(B)`.

## Агенты

### Visitor (посетитель)

**Тип:** реактивный BDI-агент.
**Количество:** настраивается (по умолчанию 65).
**Файл:** `src/agt/visitor.asl`

#### Beliefs
- `my_budget(B)` -- личный бюджет (начальный 200-500, ежедневный доход +10..+30).

#### Goals
- `!init_visitor` -- инициализация бюджета.
- `!do_one_action` -- принятие решения о визите (каждый шаг).

#### Поведение -- 3-ступенчатая воронка

Каждый день посетитель проходит три последовательных вероятностных фильтра:

```
P(визит) = P(добрался) x P(оплатил) x P(захотел)
```

1. **Транспортный фильтр**: `P(добрался) = transport_access / 100`. Если транспортная доступность упала до 40, только 40% посетителей даже доберутся до музея.

2. **Платёжный фильтр**: `P(оплатил) = payment_system / 100`. Моделирует надёжность платёжных терминалов.

3. **Фильтр желания**: `P(захотел) = (A + Satisfaction + IQ/2 - W/2) * SF / 250`, где `Satisfaction = (MN + NA + SA) / 3`. Зависит от привлекательности, качества инфраструктуры, износа и сезона.

При визите посетитель оставляет **отзыв**: `Review = (MN + SA + NA + IQ) / 4 - W/2 + noise(-10..+10)`.

30% посетителей с достаточным бюджетом бронируют **гостиницу**.

### Manager (менеджер) -- AI-powered

**Тип:** BDI-агент, стратегические решения делегированы LLM через OpenRouter API.
**Количество:** 1.
**Файлы:** `src/agt/manager.asl`, `src/java/ai/decide.java`, `src/java/ai/AIClient.java`

#### Архитектура

Менеджер разделён на два уровня:

- **Механические действия (хардкод ASL, приоритеты 0--1.5)** -- выполняются мгновенно, без обращения к LLM.
- **Стратегические решения** -- делегируются LLM через internal action `ai.decide(Action, Target)`.

#### Beliefs
- `negotiating(yes/no)` -- идёт ли переговорный процесс с реставратором.
- `negotiating_since(Day)` -- день начала переговоров (для таймаута).

#### Приоритеты решений (сверху вниз)

1. **P0: Исполнение одобренного ремонта** (ASL) -- если котировка принята и задержка = 0, выполнить `order_repair(Price)`.
2. **P0.5: Ожидание задержки ремонта** (ASL) -- обратный отсчёт до исполнения.
3. **P1: Auto-accept котировки** (ASL) -- при наличии `repair_quote` немедленно принимает.
4. **P1.5: Таймаут переговоров** (ASL) -- если ответ от реставратора не получен за 7 дней, сброс состояния.
5. **Все остальные решения** (AI) -- `ai.decide(Action, Target)` вызывает LLM.

#### AI-клиент (AIClient.java)

- **Модель:** qwen/qwen3-coder-next через OpenRouter API.
- **Недельное планирование:** один API-вызов возвращает план на 7 дней (JSON-массив из 7 действий). Следующие 6 дней берутся из кеша мгновенно.
- **Инвалидация плана:** при получении котировки, отказе реставратора или начале ремонта кеш сбрасывается и запрашивается новый план.
- **Предвычисленные подсказки (hints):** Java-код передаёт модели готовые результаты сравнений (`wear_critical`, `all_infra_above_90`, `min_infra_factor`, `max_investments` и др.), чтобы LLM не ошибалась в числовых сравнениях.
- **Graceful degradation:** при отсутствии `OPENROUTER_API_KEY` или ошибке API менеджер делает `skip`.

#### Доступные AI-действия

| Действие | Описание |
|---|---|
| `skip` | Пропустить ход |
| `invest_infra(Factor)` | Инвестировать 500 в фактор инфраструктуры |
| `invest_attractiveness` | Инвестировать 500 в привлекательность |
| `request_repair` | Запросить котировку у реставратора |
| `accept_quote` | Принять текущую котировку |
| `reject_quote` | Отклонить текущую котировку |

### Restorer (реставратор)

**Тип:** BDI-агент с экономическим поведением.
**Количество:** 1.
**Файл:** `src/agt/restorer.asl`

#### Beliefs
- `workload(0..2)` -- текущая загрузка (случайное блуждание: 30% шанс +1, 20% шанс -1 ежедневно).
- `available(yes/no)` -- доступность (15% шанс переключения ежедневно).

#### Поведение

**Ежедневное обновление состояния**: загрузка и доступность меняются стохастически, моделируя занятость реставратора на внешних проектах.

**Обработка запроса котировки** (от менеджера):
- Если `available(no)` -- отказ (`repair_refused`).
- Если `workload >= 2` -- отказ (перегружен).
- Иначе -- расчёт цены: `Price = Wear * 100 + Workload * 500`, с 20% шансом задержки на 1 день.

**Ремонт**: пока `repairing(yes)` и `wear > 20` -- ежедневно выполняет `do_repair` (wear -1%). По завершении (wear <= 20%) среда снимает оставшуюся оплату.

## Взаимодействие агентов

```
Manager ──achieve quote_request(Day)──> Restorer
Restorer ──tell repair_quote(Price,Delay,Day)──> Manager   (или repair_refused(Day))
Manager ──order_repair(Price)──> Environment
Restorer ──do_repair──> Environment (ежедневно, пока wear > 20%)
Visitors ──visit(yes/no, Review)──> Environment
Environment ──percepts──> All agents
```

Менеджер и реставратор общаются через **FIPA-сообщения** (`.send`), а не через среду. Среда выступает общим хранилищем состояния, доступным всем агентам через перцепты.

## Петли обратной связи

1. **Отзывы -> Привлекательность -> Посещаемость**: плохая инфраструктура снижает отзывы, что снижает привлекательность, что уменьшает вероятность визитов.
2. **Посещаемость -> Доход -> Инвестиции -> Инфраструктура**: больше посетителей = больше дохода = больше возможностей для инвестиций в инфраструктуру.
3. **Износ -> Ремонт -> Вместимость**: высокий износ вынуждает ремонт, который сокращает вместимость музея вдвое, снижая доход на время ремонта.
4. **Посещаемость -> Нагрузочная деградация**: больше посетителей ускоряет износ мобильной сети и платёжной системы.
